| 2026-04-17 | **Multi-part query assembly pattern (pilot-validated, subject-agnostic):** added "Answering the original request after corpus is complete" section to `research-workflow.md` and new Mandatory Rule 8 in `SKILL.md`. After the notebook is built out, don't issue one monolithic query for the final deliverable — NotebookLM's tool-result size is dominated by the citations/references payload (each cited source's `cited_text`, sometimes full `cited_table` rows). Monolithic queries against 90+-source notebooks reliably exceed the output cap even when the answer text is small. **Partition by the deliverable's outline, not by "ask everything at once."** Derive the outline from the user's original request first — shape differs by task type (tool evaluation / literature review / investigation / how-to / comparison / market analysis / biography / concept primer / policy or legal question / medical or scientific condition). Then issue 6–10 sub-queries, one per outline heading. Query-shape invariants apply regardless of topic: table-only cells or explicit word budget (~500-1000 words), bracketed citation numbers only, inherit session-scoped scope exclusions, reuse the same `conversation_id`. Always end with a gap-query sub-query ("what is NOT well-covered?"). Assemble inline in outline order. Validated on Firecrawl-Alternatives pilot: monolithic query returned 96,130 chars (cap exceeded); compact retry 85,004 chars (still exceeded); 8-query partition delivered cleanly. The partitioning axes in the pilot (candidates / archetypes / ops / failure-modes) were task-type-specific to tool evaluation — other task types partition along different axes from their own outlines. |
| 2026-04-17 | **Iterative fast-mode discovery pattern (pilot-validated):** replaced the implicit "run one deep research task per notebook" default with a multi-cycle fast-mode loop. Key rules added to `research-workflow.md` under new "Iterative Fast-Mode Discovery" section: (1) **per-request exhaustion tracking** — each `research_start` task has its own approval rate; a query is exhausted only when THAT query drops below 50%, don't average across the batch; (2) **gap-query cadence** — run `notebook_query` after every cycle starting from C2, in parallel with cycle-N polling, to seed cycle-N+1; re-run with similar wording each time (notebook content changes between cycles, so answers change); (3) **refined angles, not repeat keywords** — when a topic is still warm, re-query with a different slice (architectural → configuration → failure-mode) rather than the same keywords; NotebookLM returns near-identical results for identical queries; (4) **task-ID aliasing caveat** — a fast task launched alongside a running deep task can be aliased on `research_import` to the deep task's pool; retry once with canonical status task_id, then accept orphan and re-launch isolated; (5) **scope exclusions are session-scoped filters** — when user says "topic X resolved elsewhere, exclude here", apply to every subsequent curation prompt AND batch-delete already-imported sources matching X, then re-run gap query. Pilot: Firecrawl-Alternatives notebook b2f25fe9 over 5 cycles + 1 deep task → 94 curated sources, 4 artifacts.
